## KN18: Executor×Advisor協調構造

```
Executor（Sonnet）
  - 毎ターン自走
  - 実装・出力担当
  - 表層高速海流

Advisor（Fable5）
  - オンデマンド呼び出し
  - 深海灯台
  - 高精度・高コスト

人間
  - 軸をFable5に投げる
  - 最終判断・責任者
```

**特徴:** 非対称構造（常時稼働 × オンデマンド）  
**コスト設計:** Executor安価・高速 / Advisor高精度・要コスト  
**責任:** 人間が構造（軸）を設計する。AIは実行する

## KN19: シリーズ記事の公開順設計原則

**原則:** 読者の理解積み上げ順で公開する

```
前提知識を作る記事 → 中間記事 → 統合記事
（磁石・プロンプト）→（軸語彙）→（海モデル）
```

**理由:** 後続記事が前提として参照する概念を先に公開しないと文脈が飛ぶ  
**適用:** 思想系コンテンツのシリーズ設計全般

## KN20: Enablerとしての Fable5

- Fable5の機能: Enabler（可能にする・開く）
- E始まり語の系譜: Enable / Expand / Expose / Enlighten
- 役割: 「人間ができないこと」を「できるようにする」
- Executor（実行）とは対照的な「可能化」の立場

## KN21: フォルダ構造と概念構造の写像原則

**設計時の確認事項:**
1. フォルダ名は概念の層（操作/説明/思想）を表しているか
2. 物理整理（ファイル置き場）と概念整理を混同していないか
3. READMEで「このフォルダが何の概念層か」を明示しているか
