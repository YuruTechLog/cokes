# 第7章: 実行フローと状態モデル（実務パターン）

**目的:** Approach Axisを使った実際の作業手順と、目的・リスク・可逆性に応じた操作順序の選び方、各結果の判定基準を示す。

## 骨子

- **共通の実行フロー5段階の掲載**
  1. マッピング（人の軸レンジ × AIプロファイルを可視化）
  2. プランニング（軸語彙・操作プリミティブ・受入条件を決定）
  3. 実行（操作プリミティブを適用してAI出力を取得）
  4. Axis評価（受入条件 + 人的レビューで判定）
  5. Cokes判定と記録・Handoff（品質維持と委任可能範囲を確認し、判定ラベルと次アクションをドキュメント化）
  （参照: `axis-log-01/approach-axis-framework.md` §実行フロー（短縮））

- **操作順序の2パターン**
  - 制約先行型: 安全境界や対象範囲を先に絞ってから作業する。不可逆な処理、対象が明確な作業、逸脱コストが高い作業に向く。
  - 探索先行型: 安全境界を確保したうえで候補や観点を広げ、作業してから表現上の制約を絞る。創作、仮説検討、複数案の比較に向く。
  - 2つは固定手順ではなく、途中で切り替えられる運用パターンとする。

- **パターン選択の観点**
  目的の明確さ、失敗時の可逆性、逸脱のリスク、候補を残す価値、確認コストを記録して選択する。不可侵領域だけは探索先行型でも後回しにしない。

- **公開プロンプト集の解析例**
  OpenAI Academyなどの公開プロンプト集を、Approach Axisの検証用サンプルとして読む。用途・差し込み変数・指示文・出力形式・実行導線を分解し、Position / Magnitude / Constraint / Filter / Blendなどへ仮分類する。
  - まず、公開資料に書かれている構造を観測事実として記録する。
  - 次に、Axisの用語で説明できる部分と説明しにくい部分を分ける。
  - 最後に、元プロンプトとAxisで再構成したプロンプトを同一課題で比較し、出力の受入条件・安定性・修正回数・人間の扱いやすさで評価する。
  - 不一致が見つかった場合は、Axisへ無理に合わせず、用語のズレ・構造の不足・実務上の有効性の差として記録する。
  （参考: [OpenAI Academy「ChatGPT for any role」](https://academy.openai.com/public/clubs/work-users-ynjqu/resources/chatgpt-for-any-role)）

- **状態モデルの図を掲載**
  Idle → Plan → Execute → Axis Evaluate → Cokes判定
  Cokes判定の分岐:
  - Axis SuccessかつCokes通過（受入条件・上位整合を満たす）→ Lock → Handoff
  - Partial（受入条件の一部を満たす。修正可能）→ 再アプローチ計画
  - Failure（重大な逸脱、検証不能な主張、または不可侵領域違反）→ 停止・原因分析・不可侵強化
  （参照: `axis-log-01/approach-axis-framework.md` §状態モデル）

- **評価指標の掲載**
  判定ラベル（3種: Success / Partial Success / Approach Failure）+ 人的レビュー。
  数値スコアや普遍的な閾値は前提にしない。受入条件はタスクごとに自然言語で定義する。

- **章間の分担**
  第7章では公開プロンプトを「解析方法の実例」として扱い、個別の再利用テンプレートや用途別カードは第10章へ送る。公開資料はAxisの証明ではなく、参考となる公開資料・比較対象として扱う。
  （参照: `axis-log-01/approach-axis-framework.md` §評価指標）

- **Success/Partial/Failureの判定基準**
  Success: 人的レビュー合格、意図方向への逸脱なし、明示したConstraint・Filterを満たす
  Partial: 一部が意図から外れているが、変更箇所を特定して修正可能
  Failure: ハルシネーション、検証不能な主張、不可侵領域違反、または意図方向への完全逸脱
  （参照: `axis-log-01/approach-axis-framework.md` §状態モデル）

- **Axis評価とCokes判定の分離**
  Axisは指定方向への接近と操作上の制約を判定し、Cokesは上位体系との整合、責任、根拠、次工程の行動範囲を判定する。両方のゲートを通過して初めてLockとHandoffへ進む。

- **Handoff前提と再評価**
  Handoffには成果物だけでなく、委任する工程・任せられる範囲・成立前提・参照資料・未確定事項・再評価条件を含める。人間が定義した評価軸に沿う品質が維持され、前提が変わるまでは工程を任せられる状態をHandoff可とする。目的、対象範囲、Boundary、正本、責任者などの前提が変わったら、旧Lockを無効化し、Axis評価とCokes判定を必要な範囲でやり直す。
  「任せられるか」の評価と「任せる」という委任決定は分ける。委任可能性を確認しても、実際に委任するか、人間の監視を残すかは人間が決める。

> **[確定]** 品質スコア・閾値（90/70）は採用しない。主観軸の定量化ではなく、3段階ラベル・受入条件・人的レビューで判定する。
