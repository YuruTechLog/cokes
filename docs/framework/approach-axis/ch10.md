# 第10章 | テンプレート集と実践Tips

> 本文 v1 / 2026-07-23（レビュー済み・昇格済み）
> 参照: `glossary.md`、`docs/knowledge/axis-log-01/approach-axis-framework.md`、`docs/knowledge/axis-log-02/integrated_knowledge.md`、`docs/knowledge/axis-log-03/integrated_knowledge.md`、`docs/knowledge/axis-log-04/integrated_knowledge.md`

## 10.1 この章の使い方

第1〜9章で整理した概念を、実務で記録・再利用できる最小単位へ落とす。テンプレートの項目はすべて埋める必要はないが、目的、評価軸、境界、判定、次アクションが抜ける場合は、Handoffや再利用の前提が失われる。

擬似コードはAI内部の関数・APIではない。`Position`は意味上の寄せ先、`Magnitude`は自然言語のニュアンス、`Constraint`と`Filter`は運用上の制約・選別を記録するラベルである。

（参照: `ch06.md` §6.10、`glossary.md`「操作プリミティブ」）

## 10.2 運用カード

1回のApproachを、後から比較できる形で記録するカードである。

```text
タイトル：
目的：
対象・範囲：
人間の評価軸：
不可侵領域（Constraint）：
AI Profile：

Position：例）概念説明へ寄せる
Magnitude：例）slight（少し広げる）
操作語彙・代表指示：
Filter：例）事実確認済みの候補のみ残す
Blend / Retry：

受入条件：
判定：Success / Partial Success / Approach Failure
人的レビュー要点：
次アクション：
記録日時・担当者：
```

### 記入例

```text
目的：第2章の概念説明を非専門家向けに整える
対象・範囲：本文の説明部分。実装手順は対象外
人間の評価軸：概念の正確さを保ち、読者が地形の比喩を理解できる
不可侵領域：glossary.mdの定義を変更しない
AI Profile：Sonnet。長文整理は得意だが、存在しない状態遷移を補完しないか確認する

Position：概念説明へ寄せる
Magnitude：slight（主題と構成を維持したまま少し広げる）
操作語彙・代表指示：専門用語を増やさず、隣接する例を1つだけ加える
Filter：正本と一致しない新語を除く

受入条件：主題、用語、境界が正本と一致し、例が本文の目的を逸脱しない
判定：
人的レビュー要点：
次アクション：
```

数値の品質スコアや信頼度を記録する欄を設けることはできるが、普遍的な閾値や内部確信度の代替とはしない。判定はタスク固有の受入条件、Success / Partial Success / Approach Failure、人的レビューで行う。

## 10.3 Handoffテンプレート

Handoffの定義・責務放棄の判定は第7章 §7.7 を正本とする。ここでは、その記録要件を実務向けテンプレートへ展開する。

```text
タイトル：Approach Axis Handoff — [プロジェクト] — [日付]

1. 背景と目的
   - 何を達成する工程か：
   - 対象と対象外：

2. 人間の評価軸
   - 目的に対する必要十分条件：
   - 優先する観点：
   - 不可侵領域・禁止事項：

3. AI Profileと入力地形
   - 使用モデル／ツール：
   - 既知の得意・不得意：
   - 入力、参照、glossaryの版：

4. 実施したApproach
   - Position / Magnitude：
   - Constraint / Filter：
   - Blend / Retryの変更履歴：

5. 評価結果
   - 判定：Success / Partial Success / Approach Failure
   - 受入条件との対応：
   - 人的レビュー要点：
   - 未解決事項・リスク：

6. 委任条件（Cokes判定）
   - 委任する工程：
   - 任せられる範囲：
   - 監視方法：
   - 前提変更時の再評価条件：
   - 責任者：

7. 次アクション
   - 次工程：
   - Lockの対象：
   - 再アプローチ条件：

8. 履歴
   - 作成・変更日時：
   - 判断者：
   - 前提変更と再評価の履歴：
```

テンプレートを埋めても、Handoff可否は第7章 §7.7 の二段階ゲートで判定する。

（参照: `glossary.md`「Handoff」、`ch07.md` §7.7）

## 10.4 日本語で操作するときの3原則

日本語での操作では、誰が何をするのか、文の核心は何か、どの方向へ寄せるのかを明示する。

1. **主語明示** — 「私が」「AIが」「読者が」を省略しない。
2. **文末核心語彙** — 最重要の概念・判断を文末に置く。
3. **軸語彙前置** — 応答方向を決める語彙を文頭付近に置く。

例：

```text
AIが、非専門家の読者に向けて、概念の正確さを保ったまま、少し具体例を広げて説明する。
```

これは日本語が他言語より優れているという主張ではなく、主語省略・同音異義語・文脈前提が操作の境界を曖昧にしやすい場合の実務上の補助である。

（参照: `docs/knowledge/axis-log-04/integrated_knowledge.md` §T03）

## 10.5 画像生成への適用例

画像生成でも、構造を先に固定し、雰囲気を後から調整するという順序を実践例として使える。

| 段階 | 操作対象 | 例 |
|---|---|---|
| Layer 1 Skeleton | 構図・ポーズ・配置 | ControlNet等で物理的な構造を固定する |
| Layer 2 Material & Lighting | 素材感・光源 | Prompt Schedulingの前半で指定する |
| Layer 3 Aesthetic Modulation | 画風・スタイル | LoRAやPrompt Schedulingの後半で調整する |

```text
構造語彙 → 素材・光源語彙 → 雰囲気・画風語彙
```

構造が未確定のまま画風やLoRAを先に強く適用すると、Layer 1の不整合を見えにくくしたまま出力が崩れる可能性がある。まず構造を確認し、次に素材・光源、最後に雰囲気を調整する。

Prompt Schedulingのような実装固有の重み構文を使う場合、その数値はツール側の構文であり、Axisの`Position`や`Magnitude`の内部強度を表すものではない。この区別は明示する。

（参照: `docs/knowledge/axis-log-04/integrated_knowledge.md` §T04・§I03）

## 10.6 Executor×Advisorのパーツ単位発注

作業を一度に丸ごと渡さず、骨格・部品・評価を分けて発注する。

```text
人間：目的・評価軸・必要十分条件・転換点・疑問点を定義
  ↓
Advisorモデル（例：Fable 5）：軸を保持し、作業を分解して発注
  ↓
Executorモデル（例：Sonnet）：担当部品を生成・整形・変換
  ↓
Advisor：軸・境界・受入条件を照合
  ↓
Cokes：品質維持と委任可能性を判定
  ↓
人間：必要な承認・監視・最終判断
```

Advisorモデルを置く価値は、生成品質の高さだけではない。人間との意識合わせ、軸保持、作業分解、Executorの結果評価によって、人間の逐次確認負荷を圧縮できるかで判断する。効果が確認できない場合は、安価なモデルを直接Executorとして使う方が合理的である。

### 高性能Advisorの補助的な使い方

人間が作った骨格に対して、考慮漏れ候補、反証、例外ケース、別解を広く列挙させる。これは探索範囲を広げるための補助であり、網羅性や正しさを保証する機能ではない。候補の採否、評価軸、不可侵領域は人間が保持する。

（参照: `docs/knowledge/axis-log-02/integrated_knowledge.md` §KN18、`docs/knowledge/axis-log-03/integrated_knowledge.md` §IK09、第9章 §9.4）

## 10.7 ツールの組み合わせ

作業の性質に応じて、クラウドモデル、ローカル生成器、コード・自動化ツールを組み合わせる。AIを使うこと自体を目的にせず、決定権と再現性が必要な箇所を人間またはコード側に残す。

| 作業 | 例 | 決定権の置き場所 |
|---|---|---|
| 推論・分析 | クラウドLLM | 人間の評価軸と受入条件 |
| 画像素材 | ローカル画像生成 | 構造・採否の人間判断 |
| バナー合成・文字配置 | Python / Playwright等 | コードで固定し、AIは素材生成に限定 |

この表は特定ツールの推奨ランキングではない。データ、コスト、再現性、権限、レビュー可能性を工程ごとに比較する。

## 10.8 成果物の保持とライフサイクル

Approach Axisの成果は、会話の中で一度出力されて終わるものではない。実務では、次のような異なる形式の成果物になる。

| 成果物 | 主な役割 | 保持すべきもの |
|---|---|---|
| Prompt | AIへの入力・操作手順 | 本文、変数、適用条件、使用したAxis Words、禁止事項 |
| Skill | 単一責任の再利用可能な作業単位 | 入力、出力、処理手順、失敗条件、受入条件、依存Skill |
| PGコード | 決定的・検証可能な処理 | ソース、テスト、依存関係、権限、実行方法、ロールバック方法 |
| Workflow | 人間・AI・ツールの工程順序 | 状態、分岐、担当、承認ゲート、Retry条件、Handoff条件 |
| Pipeline | 複数工程を連結した実行系 | 構成、入出力契約、実行環境、監視、再実行、停止条件 |

### 成果物に共通するメタデータ

実行物だけを保存すると、後から「なぜこの形になったか」「どこまで任せてよいか」が分からなくなる。最低限、次のメタデータを成果物と同じ版に紐づける。

```text
成果物ID：
種類：Prompt / Skill / PGコード / Workflow / Pipeline
目的・適用範囲：
入力・出力契約：
依存関係・実行環境：
参照した正本・glossary版：
使用したAxis Words / Position / Magnitude / Constraint / Filter：
AI Profile・Advisor / Executor構成：
受入条件・テスト：
既知の失敗・未解決事項：
責任者・レビュー者：
版・変更履歴：
Handoff範囲・前提変更時の再評価条件：
状態：Draft / Reviewed / Locked / Released / Superseded / Retired
```

### 保持場所の分離

保持場所は、役割ごとに分ける。具体的なディレクトリ名はプロジェクトのリポジトリ規約に従うが、次の分離を崩さない。

1. **正本リポジトリ** — Prompt、Skill、コード、Workflow、Pipelineのレビュー可能なソース。Git等で版管理する。
2. **実行パッケージ** — 実際に利用する固定版。正本のコミットや成果物IDへ戻れるようにする。
3. **評価・運用記録** — 入力条件、出力要約、判定、人的レビュー、Retry、Handoff、前提変更を記録する。秘密情報や個人情報は実行物と分離する。
4. **成果物カタログ** — ID、種類、所有者、状態、依存、最新版、廃止版、再評価条件を検索できるようにする。

会話ログは判断の経緯を補助する記録であり、成果物の正本そのものにはしない。採用したPromptやコード、評価条件を正本側へ移し、再現可能な版として固定する。

### 推奨する初期構成（案）

リポジトリに置く場合は、種類と状態を分けて検索できる最小構成から始める。

```text
artifacts/
  prompts/       # Prompt本文・変数・適用条件
  skills/        # 単一責任のSkill
  code/          # PGコード・テスト・依存定義
  workflows/     # 工程・分岐・承認ゲート
  pipelines/     # 複数工程の実行定義
  manifests/     # 成果物メタデータ（ID・版・前提・責任者）
  evaluations/   # 評価・Retry・Handoff・前提変更の記録
```

実行環境の秘密情報、個人情報、巨大な生成ログはこの構成へ直接混ぜず、アクセス制御された保管先から成果物ID・版・ハッシュで参照する。まずはGitでレビュー可能な正本と、実行・評価記録のリンクを作ることを優先し、管理基盤の導入は量と権限要件に応じて後から決める。

### ライフサイクル

```text
Draft
  → Reviewed（人間が目的・軸・境界・テストを確認）
  → Locked（適用範囲と前提を固定）
  → Released（実行パッケージとして利用）
  → Superseded（新しい版へ置換）
  → Retired（利用停止・再利用不可）
```

前提、正本用語、入力地形、AI Profile、依存Workflow、権限、後工程が変わった場合は、ReleasedやLockedの成果物を黙って使い続けない。影響範囲を確認し、必要なAxis評価とCokes判定をやり直して新しい版を発行する。

成果物の再利用可能性は、コードが動くことだけでは判定しない。人間の評価軸、受入条件、責任者、再評価条件まで復元できて初めて、SkillやWorkflowを安全に再利用・委任できる。

## 10.9 章の要点

- 運用カードは、自然言語のニュアンスと受入条件を記録する。
- Handoffは成果物の受け渡しではなく、委任工程・範囲・前提・監視・責任を含む条件付き出口である。
- 日本語では主語、核心語彙、軸語彙を明示する。
- 画像生成では構造→素材・光源→雰囲気の順序を実践例として扱う。
- Advisor／Executorはモデルの役割分担であり、Fableという体系を追加するものではない。
- 高性能Advisorによる考慮漏れ探索は補助例であり、網羅性の証明ではない。
- Prompt、Skill、PGコード、Workflow、Pipelineは、実行物と評価・前提・責任のメタデータを同じ版に紐づけて保持する。
- Handoff可能な成果物は、内容だけでなく、適用範囲と再評価条件を復元できる。

## 判断が必要な点

1. 運用カードで「AI Profile」をどこまで必須項目にするか。
2. 画像生成のPrompt Scheduling構文に数値を載せるか。載せる場合は、Axisの操作量ではなくツール固有構文である注記を残す。
3. `slight / moderate / strong` の日本語ラベルを、テンプレート上で英語併記するか。
4. Executor×Advisorの比較指標（軸保持率・見逃し率・レビュー率・総コスト）を、テンプレートに固定項目として加えるか。
5. 成果物カタログをリポジトリ内のMarkdown／YAMLで持つか、外部の管理基盤へ委ねるか。

## 用語対応

| 本章の表現 | 正本用語 |
|---|---|
| 寄せ先 | Position |
| 寄せ方のニュアンス | Magnitude |
| 不可侵領域 | Constraint / Boundary |
| 候補の選別 | Filter |
| 条件付きの工程委任 | Handoff |
| 上位モデルの軸保持役 | Advisorモデル（Fable 5は例） |
| 下位モデルの実行役 | Executorモデル（Sonnetは例） |
| 成果物の版・前提・評価をまとめた記録 | 成果物メタデータ（追加候補） |

用語の定義は `glossary.md` を優先する。外部資料・研究・ツール構文は参考として扱い、Axisの内部実装や普遍的な有効性を直接証明するものとは書かない。
