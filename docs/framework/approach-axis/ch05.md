# 第5章: 軸語彙リファレンス

> 本文 v1 / 2026-07-23（レビュー済み・昇格済み）  
> 用語の正本: `glossary.md`

第4章では、Approach Axisを人間とAIの間で運用する4層を整理した。本章では、その第1層である**軸語彙層（Axis Words）**を参照用に整理する。

ここで扱うのは、AI内部の構造を直接読んだり、特定モデルの性能を順位づけたりする方法ではない。人間がどの方向へ意味を寄せたいかを言葉にし、意味地形図をどのように傾けるかを設計するための作業上の分類である。

軸語彙は磁石ではない。軸語彙が地形に傾きをつけ、プロンプトという磁石がその傾斜を転がりながら意味を巻き込む。傾斜が強すぎれば磁石に関係なく砂が落ち、Approach Failureになる。この関係は第2章の地形磁力モデルに従う。

## 5.1 5軸と補助軸

5軸は、人間の判断を異なる種類の操作へ分けるための分類である。順序（Sequence）は、5軸を実行する順番を補助するもので、独立した第6軸として数えない。ここでいう「補助」は効力が弱いという意味ではなく、他の軸が意味の方向・範囲・粒度を指定するのに対し、Sequenceは提示・処理順を指定するという機能上の区別である。

| 分類 | 英語名 | 役割 | 例語彙 |
|---|---|---|---|
| 起点 | Intent | 思考・作業の出発点を固定する | 「Aを起点に」「目的はA」 |
| 抽象度 | Abstraction | 扱う粒度や説明の高さを調整する | 「概念レベルで」「具体例まで」 |
| 境界 | Boundary | 対象範囲と不可侵領域を限定する | 「この範囲だけ」「Aは除く」 |
| 優先順位 | Priority | 条件が衝突したときの優先順を示す | 「最優先はA」「BよりA」 |
| 方向性 | Direction | 出力を寄せる解釈・表現の方向を示す | 「Aの視点で」「A方向に寄せて」 |
| 順序（補助） | Sequence | 複数の軸や作業を段階化する | 「まずA、次にB」 |

分類は語彙の辞書ではなく、今回の操作で何を固定するかを選ぶための枠である。同じ語が複数の役割を持つ場合もあるため、語そのものより、文中で担わせた役割を記録する。

### 起点（Intent）

何を達成したいのか、どの問いから考えるのかを定める。目的が曖昧なまま他の軸を強めると、別の意味領域へ傾斜がつく。起点は、回答の目的と対象を一文で置くと扱いやすい。

**例:** 「第4章の4層構造を、初めて読む人が運用手順として理解できるように説明する」を起点にする。

### 抽象度（Abstraction）

概念・原理・設計・手順など、どの高さで扱うかを調整する。抽象度を上げると個別例をまとめやすくなる一方、実行条件が抜けやすい。下げると具体的な作業へ近づくが、局所例に引っ張られやすい。

**例:** 「概念レベルで説明し、具体的なプロンプト構文は扱わない」と指定して、実装手順への落下を抑える。

### 境界（Boundary）

何を対象にし、何を対象外にするかを決める。不可侵領域（Constraint Zone）や出力形式も、必要に応じて境界として明示する。境界を強めるほど範囲外の砂は落ちにくくなるが、狭すぎると必要な前提まで排除する。

**例:** 「外部理論は参考となる学術理論として紹介するが、Approach Axisの実装証明とは扱わない」と対象外の解釈を明示する。

### 優先順位（Priority）

複数の条件が同時に成立しないとき、どれを先に守るかを示す。優先順位だけで目的を代替しようとすると、局所的な最適化になりやすい。起点と境界を置いた後に、衝突解消のために使う。

**例:** 「用語集との整合を、読みやすさや新しい言い換えより優先する」と決める。

### 方向性（Direction）

どの解釈・観点・表現へ出力を寄せるかを示す。方向性は地形全体の傾きを変える語彙として働きやすいが、境界がない場合は広い意味領域を巻き込みやすい。方向を明示した後、境界と抽象度で範囲を整える。

**例:** 「人間とAIのインターフェースとして解釈する方向へ寄せる」と指定し、単なるモデル内部の説明へ流れるのを防ぐ。

### 順序（Sequence・補助）

複数の軸語彙を一度に競合させず、段階的に提示するための補助である。典型的には「起点・方向性 → 境界 → 抽象度 → 優先順位 → 実行」の順に置く。ただし、危険領域を先に除外する必要がある場合は境界を先行させる。

**例:** 「まず用語集の定義を確認し、次に章の目的を置き、その後に具体例を書く」と段階を指定する。

## 5.2 軸語彙を組み合わせる

一つのプロンプトにすべての軸を最大強度で詰め込むと、地形の傾斜が強くなりすぎ、意図と無関係な意味まで落ちる。まず今回の目的に必要な軸を選び、軸同士の役割を分ける。

```text
起点・方向性を置く
  → 境界で範囲を限定する
  → 抽象度で粒度を合わせる
  → 優先順位で衝突を解く
  → 順序に沿って実行・評価する
```

例えば「研究上の視点を参考にしつつ、Approach Axisの説明として書く」場合は、次のように役割を分ける。

```text
起点: Approach Axisの読者向け説明を作る
方向性: 人間とAIのIFとして解釈する
境界: 外部理論を実装証明として扱わない
抽象度: 概念説明の高さに留める
優先順位: 用語集の定義を最優先する
順序: 定義 → 比喩 → 注意書き → 例
```

この例で「外部理論を実装証明として扱わない」は、研究層の知見を安全に解釈層へ渡すための境界である。軸語彙を増やすことより、各語彙の責務を重複させないことが重要になる。

## 5.3 軸語彙の効力傾向

旧資料では「磁力強度マップ」と呼んでいたが、現行の**意味地形図（Semantic Terrain Map）**とは役割が異なる。意味地形図は意味の配置・高低・近接関係を読むための層であり、ここで扱うのは、軸語彙がその地形へ与える影響の傾向である。本章では、これを「軸語彙の効力傾向」と呼ぶ。

初期の作業仮説は次の順序である。

```text
方向性 > 境界 > 抽象度 > 起点 > 優先順位 > 順序
```

これは全モデル共通の法則でも、定量的な強度スコアでもない。学術文献がこの並びを直接実証したものでもなく、Approach Axis独自の作業ヒューリスティクスである。タスクの抽象度、境界の厳しさ、AIプロファイル、プロンプトの粒度によって入れ替わる。順序は意味の向きを変える軸ではなく、複数の軸を衝突させないための補助なので、単純な強弱比較には向かない。ただし、補助であることは影響が小さいことを意味しない。提示位置や順序が出力を大きく変える場合もある。

| 軸 | 効力が現れやすい場面 | 逆転・失敗の条件 |
|---|---|---|
| 方向性 | 出力の観点・解釈を大きく寄せたいとき | 境界が弱いと、目的外の意味まで巻き込む |
| 境界 | 対象外・不可侵領域を明確にしたいとき | 狭すぎると必要な前提を落とす |
| 抽象度 | 概念説明と具体手順を切り替えたいとき | 高すぎると実行条件が抜け、低すぎると局所化する |
| 起点 | 問い・目的を固定したいとき | 目的だけで範囲を指定しないと、解釈が拡散する |
| 優先順位 | 条件や評価軸が競合するとき | 上位目的の代わりに使うと局所最適化になる |
| 順序 | 複数の指示を段階的に渡すとき | 順序だけでは意味の方向や範囲を定められない |

検証するときは、同じ課題で軸語彙を一つずつ変え、出力の差を人間が判定する。判定はSuccess / Partial Success / Approach Failureで記録し、強度の数値化は必須としない。

### 5.3.1 研究との対応範囲

関連研究は、軸語彙の分類や順位を直接証明するものではない。研究対象の層を分け、Approach Axisでは「参考となる学術理論・研究上の視点」として扱う。

| 軸・論点 | 参考となる研究上の視点 | Approach Axisでの扱い |
|---|---|---|
| 起点（Intent） | Instruction TuningやInstructGPTは指示追従を扱う | プロンプト内の起点（Intent）の単独効果を直接測定した研究とは区別する |
| 抽象度（Abstraction） | Scale-Spaceは多重解像度の理論だが、自然言語プロンプトの抽象度を直接検証する理論ではない | 抽象度を説明する比喩・研究上の視点。直接根拠とはしない |
| 境界（Boundary） | Instruction Hierarchyは、system・user・tool等の指示源に優先順位を与える | 対象範囲・不可侵領域としての境界とはレイヤーが異なる。間接的な参考に留める |
| 優先順位（Priority） | Instruction Hierarchyは、指示源の優先順位と競合時の扱いを研究する | Axisの条件間の優先順位とは対象レベルが異なる。同一視せず、間接的な参考に留める |
| 方向性（Direction） | Activation Engineering / Representation Engineeringは、モデル内部の表現方向を操作する | 人間がプロンプトで出力方向を指定する操作との類比。内部介入と同一視しない |
| 順序（Sequence・補助） | *Lost in the Middle* や few-shot prompt order研究は、提示位置・順序による出力変化を調べる | Sequenceの影響を示す傍証。順位の最下位や「弱い軸」を証明するものではない |
| 総合順位（5軸＋Sequence） | 6分類間の効力を同一課題で比較した研究は確認できない | Approach Axis独自の作業ヒューリスティクス。検証・反証可能な仮説として記載する |

ここでの「直接」「間接」は、論文の価値の順位ではなく、Approach Axisの操作語彙との距離を示す。本章は研究結果を実装証明へ変換しない。

## 5.4 AI Profileでモデル差を読む

同じ軸語彙でも、意味地形の配置と砂の合金具合が異なれば、効力の現れ方は変わる。第4章のAI特性層では、少なくとも次の4つを確認する。

| AI Profileの項目 | 確認すること | 軸語彙への影響 |
|---|---|---|
| チャンキング粒度 | 意味をどの粒度で処理するか | 抽象度・順序の効き方、文脈の切れ目 |
| 重みづけ思想 | helpfulness / harmlessness / factuality等の優先傾向 | 起点・優先順位の衝突時に残る意味 |
| 安全装置チューニング | 拒否・フィルタリングの強度と方針 | 境界を越えたときの崩壊閾値 |
| 日本語特性 | トークン効率、スクリプト混在、実効文脈長 | 日本語での粒度・近接関係の見え方 |

製造元やモデル系列による違いは、固定的な「最強軸」ではなく、検証を始めるための仮説として置く。ホスト製品のシステム指示、選択モデル、更新時期、追加の安全層によっても結果は変わる。

| 利用形態・モデル系列 | 最初に試す仮説 | 必ず確認すること |
|---|---|---|
| Claude系 | 抽象度・原則との整合 | 具体指示と原則が衝突したときの境界・拒否 |
| GPT系 | 起点・目的の明示 | 目的を固定した後の境界・方向性 |
| Copilot環境 | 優先順位・局所差分 | ホスト製品の文脈、システム指示、選択モデル |
| Gemini系 | 境界・タスク分割 | 長い文脈での境界保持と順序 |
| OSS・ファインチューン系 | 固定仮説を置かない | ベースモデル、追加チューニング、安全層、日本語データ |

この表は公開資料と実機観察を整理するための参考表であり、製造元の設計思想やモデルの内部実装を証明するものではない。外部の論文・技術レポート・モデルカードは、用語や観察項目を考えるための**参考となる学術理論・研究上の視点・公開資料**として扱う。

## 5.5 合金比率を読む

合金比率（Alloy Ratio）は、意味の砂のうち、プロンプトという磁石にひっつきやすい成分と、ひっつきにくい成分の配合を説明する比喩である。軸語彙は磁石ではなく、磁石が転がる地形の傾きを操作する。

実務では、合金比率を製造元の性格として断定せず、次の観察結果から推定する。

1. 同じ起点・方向性・境界を、複数のAIへ同じ粒度で渡す
2. どの意味が残り、どの意味が落ちたかを比較する
3. 差が重みづけ、透明性、安全装置、日本語特性のどれで説明できるかを記録する
4. 次回のAIプロファイルへ観察結果を反映する

「製造元の思想は世代を跨いで必ず継承される」とは置かない。継承される可能性は仮説として記録し、モデル世代・提供形態・システム指示が変わった時点で再確認する。

## 5.6 磁石サイズを読む

磁石サイズ（Chunk Size）は、チャンキング粒度（Chunking Granularity）を説明する比喩である。ここでいう磁石はプロンプトであり、軸語彙そのものではない。

- **大きい塊**：広い意味領域を一度に巻き込む。文脈は保ちやすいが、意図以外も付着しやすい
- **小さい塊**：意味の切れ目を細かく指定する。狙いは絞りやすいが、文脈が切れ、引き寄せる材料が減りやすい
- **適切な塊**：対象の意味領域と指示の粒度が合い、必要な砂だけが付着する

最適サイズはモデルだけで決まらない。タスクの長さ、日本語・英語などの言語、境界の厳しさ、評価単位によって変わる。表現学習やAttention機構に関する研究は、この比喩を理解する足場にはなるが、磁石・コサイン類似度・Axisの実装が同一だという意味ではない。

## 5.7 実務での参照手順

1. **AIプロファイルを記録する**：モデル系列、提供形態、言語、既知の安全層、前回の失敗を残す
2. **起点と方向性を一つずつ置く**：今回寄せたい目的と観点を短くする
3. **境界を置く**：対象外、不可侵領域、外部参照の扱いを明示する
4. **抽象度を合わせる**：概念説明か具体手順かを選ぶ
5. **優先順位と順序を付ける**：競合する条件を並べ、段階的に実行する
6. **出力を判定する**：Success / Partial Success / Approach Failureを記録し、必要なら軸語彙・粒度・AIプロファイルの仮説を更新する

この手順は、人間の軸レンジを保ったままAIの応答空間を調整するためのものだ。モデル選択やプロンプトの「呪文」を固定するための手順ではない。

## 5.8 合金×粒度で失敗を診断する

| 観察された状態 | 仮説 | 次の調整 |
|---|---|---|
| 目的外の意味まで一気に落ちる | 傾斜が強すぎる、または磁石の塊が大きすぎる | 方向性を弱め、境界を追加し、指示を分割する |
| 目的の意味がほとんど動かない | 対象成分の磁性が弱い、または塊が小さすぎる | 起点・方向性を言い換え、前提を少しまとめる |
| 途中で拒否・停止する | 安全装置チューニングが境界を越えた | 不可侵領域を確認し、目的を安全な範囲へ再設計する |
| モデルを替えると結果が変わる | AIプロファイルの差 | 4項目を比較し、効力傾向を再評価する |

「すべて落ちる」状態は、単なる軸語彙の不足ではなくApproach Failureの可能性がある。停止し、根拠トレースと不可侵領域の再確認を行ってから再設計する。

## 5.9 まとめ

- 5軸は、起点・抽象度・境界・優先順位・方向性であり、順序は補助である
- 5軸の効力順位は学術的な普遍則ではなく、Approach Axis独自の作業ヒューリスティクスである
- Sequenceの「補助」は役割上の区別であり、出力への影響が小さいことを意味しない
- 軸語彙の効力は条件依存で、固定的な強度ランキングではない
- 意味地形図は意味の配置を読む層、AI Profileはモデル差を観察する層である
- 合金比率と磁石サイズは、重みづけ思想とチャンキング粒度を説明する比喩である
- 実務では、仮説を置き、同一課題で比較し、人的判定でAIプロファイルを更新する
