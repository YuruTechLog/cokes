# 第8章 | 失敗モードと診断

> 本文 v1 / 2026-07-23（レビュー済み・昇格済み）
> 参照: `glossary.md`、`docs/knowledge/axis-log-01/approach-axis-framework.md`、`references/terrain-magnetic-model.md`、`references/model_comparison_axis.md`、`docs/knowledge/axis-log-03/integrated_knowledge.md`

## 8.1 この章の役割

Approach Axisは、出力を人間の評価軸へ近づける操作だけでなく、近づけることに失敗したときの停止・診断・再設計までを含む。第7章の状態モデルで `Partial` または `Failure` になった場合、この章の手順へ戻る。

ここでいう失敗は、モデル内部の原因を特定したという意味ではない。観測できる入力・出力・評価結果から、次の操作を安全に続けられない状態を判定する。

（参照: `docs/knowledge/axis-log-01/approach-axis-framework.md` §定義・§状態モデル、`glossary.md`「Approach Failure」）

HandoffとCokes側の責務放棄の定義・判定条件は第7章 §7.7 に集約する。本章では、Failureを検出した場合に既存のHandoffを停止し、再評価へ戻す。

## 8.2 Approach Failureの定義

**Approach Failureとは、軸語彙やプロンプトによる操作の結果、出力が人間の許容する軸レンジから逸脱し、現在の前提のまま継続できない状態である。**

地形磁力モデルでは、地形の傾斜が強すぎて、磁石（プロンプト）の狙いに関係なく意味の砂が一斉に落ちる「地形崩壊」として説明する。これは操作のための解釈モデルであり、LLM内部が実際にそのように実装されていることを主張するものではない。

次のいずれかを検出したら、通常のRetryではなくApproach Failureとして扱う。

- 人間が設定した不可侵領域を越えた
- 根拠のない内容、捏造、または検証不能な断定が混入した
- 目的・境界・出力形式が大きく外れ、局所修正で戻せない
- 重要な前提が変わった、または前提の不一致が判明した

品質スコアや閾値を使う場合でも、値はプロジェクトごとの補助指標にすぎない。`90/70` のような数値は素材にある例示であり、全タスク共通の基準ではない。最終判定は判定ラベルと人的レビューで行う。

（参照: `glossary.md`「Approach Failure」「判定ラベル」、`docs/knowledge/axis-log-01/approach-axis-framework.md` §状態モデル）

## 8.3 失敗モード一覧

| 失敗モード | 観測される状態 | 典型的な誘因 | 最初の対処 |
|---|---|---|---|
| Approach Failure（ハルシネーション・軸外逸脱） | 根拠のない内容、不可侵領域への侵入、目的からの大幅な逸脱 | 傾斜・指示の強すぎ、前提不一致、地形と磁石の競合 | 即停止し、根拠と前提をトレースする |
| 過剰適合 | 指示には沿うが、表現・候補・視点の多様性が失われる | Constraint / Filter の過剰、禁止事項の重複 | 制約を緩和し、複数案を取得して人的評価する |
| コスト超過 | Retry・レビュー・修正が増え、工程全体の負担が許容範囲を超える | 軸の分解不足、閾値の過剰な厳格化、モデル選択の不一致 | 反復を止め、工程分割・モデル変更・仕組み化を検討する |
| 安全装置との衝突 | 依頼の一部または全体が拒否され、目的に必要な情報が得られない | 人間の目的とモデル側の境界・安全方針の競合 | 不可侵領域を回避して目的レベルで再設計する。回避不能な境界は受け入れる |
| 前提ドリフト | 同じ操作でも評価軸、参照、対象範囲が変わり、過去の成功を再利用できない | 仕様・用語集・入力地形・担当者・後工程の変更 | 既存のLock/Handoffを無効化し、前提確認からやり直す |

「安全装置との衝突」は、モデルの拒否を失敗と断定するための分類ではない。目的達成に必要な範囲と、受け入れるべき不可侵領域を人間が分けて記録するための診断項目である。

（参照: `docs/knowledge/axis-log-01/approach-axis-framework.md` §失敗モードと対処、`glossary.md`「安全装置チューニング」「前提条件」）

## 8.4 即停止プロトコル

Approach Failureを検出したら、次の順序を崩さない。

1. **即停止** — 現在の出力を成功扱いせず、下流へのHandoffを止める。
2. **根拠トレース** — 入力、参照、Axis Words、プロンプト、AI Profile、直前の変更を記録する。
3. **不可侵領域の再強化** — 人間が譲れない条件、禁止範囲、出力の受け手を明示し直す。
4. **再設計** — 軸語彙・Prompt・Constraint・Filter・順序・モデル選択を見直し、別のアプローチとして再実行する。

停止中は、失敗出力をそのまま要約・変換・次工程へ転送しない。根拠トレースで再利用可能な部分が確認できた場合のみ、人間の判断を経て新しい入力へ組み込む。

（参照: `docs/knowledge/axis-log-01/approach-axis-framework.md` §Approach Failure（ハルシネーション）の扱い）

## 8.5 診断の観測項目

診断は内部ベクトルを推測する作業ではなく、操作と出力の対応を比較する作業である。最低限、次を1組として残す。

| 項目 | 記録内容 |
|---|---|
| 前提 | 目的、対象範囲、参照資料、glossaryの版、担当者、後工程 |
| 操作 | Axis Words、Positionのニュアンス、Constraint、Magnitude、Filter、順序 |
| 入力 | 実際に渡したPromptと、モデル・ツールの設定 |
| 出力 | 逸脱箇所、根拠の有無、欠落、拒否、過剰な固定化 |
| 判定 | Success / Partial Success / Approach Failure と人的レビュー要点 |
| 変更 | 直前のRetryとの差分、変更した前提、再評価が必要か |

同じ失敗が続く場合、同じPromptを強く繰り返すのではなく、失敗が「方向」「境界」「抽象度」「起点」「優先順位」「順序」のどこに現れたかを切り分ける。原因の断定ではなく、次の操作を選ぶための診断である。

## 8.6 モデル・製造元差の扱い

モデルごとに、軸語彙への反応、拒否の出方、文脈保持、言語・トークン効率が異なる可能性がある。したがって、失敗モードの傾向をAI Profileへ記録し、製造元名やモデル名だけで一般化しない。

| 観察対象 | 作業仮説としての傾向 | 操作上の確認 |
|---|---|---|
| Anthropic系 | 具体指示と原則が競合した場合、「効かなくなる」「拒否する」形で現れる可能性 | 目的レベルの指示と不可侵領域を分ける |
| OpenAI系 | 起点が曖昧な場合、汎用的な方向へ散る可能性 | 目的・対象・出力形式を先に固定する |
| Google系 | 意図方向や推論深度の指定が強すぎる場合、発散する可能性 | 境界を明示し、タスクを分割して比較する |
| Microsoft Copilot | 使用モデルとM365側のポリシーに依存する | 実際のモデル、DLP、感度ラベル等の制約を確認する |
| Meta系OSS | ファインチューニングと外部安全装置で挙動が変わる | 配布モデル・追加調整・安全コンポーネントを記録する |

この表は固定的な製造元ランキングではなく、検証を始めるための観察仮説である。モデルの世代、設定、入力言語、接続ツールによって結果は変わる。「測れない」と「予測できない」を同一視せず、同一条件での比較記録を優先する。

合金比率（磁性成分と非磁性成分の配合）は、意味が磁石に巻き込まれやすいかを説明する補助概念であり、地形の傾斜とは別である。モデル差を説明する比喩として使うが、学習重みの直接測定値とは扱わない。

（参照: `references/model_comparison_axis.md`、`references/terrain-magnetic-model.md` §モデル別崩壊特性・§合金比率への接続）

## 8.7 AIによるフレームワーク誤認

AIにCokesやAxisの説明・章立てを生成させた場合、出力された説明を正本とみなさない。観察済みの「Axisに存在しないプランフェーズ」を出力した事例のように、もっともらしい用語や状態を補完することがある。

対策は次の通り。

- 正本（`glossary.md` と承認済み章）を先に明示する
- 固定事実・仮説・TODOを分けて渡す
- 出力内の新しい用語や状態遷移を、正本と照合する
- 照合できないものは採用せず、必要なら人間に確認する

これはAIが悪意を持つという話ではなく、文脈から妥当そうな構造を補完する性質への対処である。

（参照: `docs/knowledge/axis-log-03/integrated_knowledge.md` §IK12）

## 8.8 Ch7への復帰とCokes判定

失敗から復帰する際は、局所的なRetryで済むか、計画全体を組み直すかを分ける。

```text
Partial
  → 変更範囲が局所的なら Retry
  → 目的・境界・前提が変わったなら再計画

Approach Failure
  → 即停止・根拠トレース
  → 不可侵領域と前提を再確認
  → Axisを再設計
  → Axis評価
  → Cokesが品質維持・委任範囲・責任境界を再判定
  → Lock / 条件付きHandoff または再停止
```

Approach Failureが一度起きた工程は、以前のLockやHandoffを自動的に復活させない。Handoffの再発行条件と責務放棄の判定は、第7章 §7.7 の記録・再評価条件に従う。

## 8.9 章の要点

- Approach Failureは、内部メカニズムの断定ではなく、許容軸からの出力逸脱として判定する。
- Failure時は即停止し、根拠・前提・操作をトレースしてから再設計する。
- 過剰適合、コスト超過、拒否、前提ドリフトも、工程を継続できない形で現れれば診断対象になる。
- モデル別特性は観察仮説。製造元・世代・設定・言語の差を記録し、普遍則として扱わない。
- Failure後のLock/Handoffは無効。Axis再評価と第7章 §7.7 のHandoff条件を経て初めて再開できる。

## 用語対応

| 本章の表現 | 正本用語 |
|---|---|
| 軸外逸脱・地形崩壊 | Approach Failure |
| 失敗からの再試行 | Retry |
| 許容できない範囲 | 不可侵領域 / Boundary |
| モデル・製造元差 | AI Profile / 合金比率 |
| 条件付きの再委任 | Handoff |

用語の綴り・定義は `glossary.md` を優先する。外部論文・技術資料は、モデル差や失敗を理解するための「参考となる学術理論・研究上の視点・公開資料」であり、Approach Axisの実装や有効性を直接証明するものではない。
