## 1. 総評

Cokes + Approach Axisの中核思想は、現時点で大きく破綻していない。

特に、次の分離は合理的である。

- Cokesは、利用手段の選択、Key-Quality、人間の責任、総コスト、Handoffを扱う
- Approach Axisは、AIを利用手段として選んだ場合の出力方向の操作・評価を扱う
- 外部研究は直接的な証明ではなく、参考となる学術理論・研究上の視点として扱う
- プロンプト操作と内部アクティベーション操作を同一視しない

ただし現状は、**よく整理された実践仮説**であり、効果を反証可能な形で確認した工学体系ではない。

最大の課題は、失敗の存在ではなく、何が起きても「Axisが効いた」「人間が判断した」「モデル差だった」と事後説明できてしまうことである。

したがって、今後の中心課題は、新しい概念の追加ではなく、次の3点になる。

1. 定義・比喩・内部機構仮説の分離
2. 成功・失敗を事前に判定できる検証設計
3. Cokes導入前後の品質・人間負荷・総コストの比較

## 2. 重要度別の指摘

### P0-1. 機構非依存と内部機構の断定が同居している

第1章とframeworkでは、Approach AxisがAI内部の機構を特定しないことを明記している。

一方、Axis用語集には次の記述がある。

- AI内部に基準軸は存在しない
- 軸語彙が確率密度を偏らせる
- 意味空間は確率密度の分布である
- BPEトークン境界で意味が切れる

これらは解釈モデルではなく、AI内部に関する事実命題として読める。

特に「内部に存在しない」「確率密度を操作する」は、機構非依存の方針を越えている。

#### 修正方針案

守る主張を次の範囲に限定する。

> Approach Axisは、AI内部に人間と同じ名前付き判断軸が存在することを前提にしない。観測可能な入力と出力の関係を操作対象とする。

地形、傾斜、確率密度、距離操作は、次のいずれかを明記する。

- 操作用の比喩
- 研究上の仮説
- 外部研究との類比
- 観測済みの出力傾向

内部機構の確定表現にはしない。

#### Terra対応

- 表現候補の抽出と一括分類: 対応可能
- 「存在しない」を「前提にしない」へ弱める修正文案: 対応可能
- どこまでをCokes / Axisの公式見解にするか: 人間判断が必要

---

### P0-2. Handoff時の責任移転がCokes正本と矛盾している

Cokes正本では、人間が目的、前提、リスク受容、最終責任を保持する。

一方、Axis用語集のHandoffには、次の表現が残っている。

- 責任を移転する
- 次フェーズへ責任が移る
- ゲート通過後に手離れし、責任移転へ進む

同じ節には「最終判断の責任は人間に残す」とも書かれているため、内部矛盾である。

#### 修正方針案

責任を最低限、次へ分解する。

| 責任・責務 | Handoff後の扱い |
|---|---|
| 実行責務 | 委任可能 |
| 定型的な評価・監視 | 条件を定義できれば委任可能 |
| 受入条件の設定 | 人間が保持 |
| 前提・リスク受容 | 人間が保持 |
| 最終的な説明責任 | 人間が保持 |

「責任移転」ではなく「実行責務の委任」「工程の手離れ」を基本表現とする。

#### Terra対応

- root `glossary.md` とAxis `glossary.md` の矛盾箇所抽出: 対応可能
- 語感を揃えた置換案: 対応可能
- 責任モデルの最終採用: 人間判断が必要

### P0-3. Axis Successが循環定義になりやすい

現行では、Position操作は「指定した方向へ寄れば成功」とされ、最終的には人間が出力を確認する。

同じ人が次を行う場合、期待効果を知った状態で自己採点することになる。

1. 軸を設計する
2. Axis版プロンプトを見る
3. 出力を評価する

これでは「なんとなく効いた呪文」をAxis語彙で追認する危険がある。

#### 修正方針案

- 受入条件を出力取得前に記録する
- 通常プロンプトとの対照を置く
- 評価時にはプロンプト種別を隠す
- A/Bの表示順をランダム化する
- 複数評価者の一致率を確認する
- 同じ意味を持つ複数の言い換えで再現性を確認する

#### Terra対応

- 検証テンプレート作成: 対応可能
- 評価用データセットと比較表の生成: 対応可能
- ブラインド評価の実施・人間評価の代行: 単独では不可

### P1-1. Key-Qualityと人間負荷の許容量が未操作化

Cokesは、目的に必要十分なKey-Qualityを満たし、人間負荷が許容量を超えないことを求める。

ただし、次が案件ごとの人間判断へ戻されている。

- 誰が必要十分と判断するか
- 未知の欠落をどう扱うか
- 人間負荷がいつオーバーフローしたと判定するか
- 品質、負荷、コストが競合した場合にどう戻るか

#### 修正方針案

「必要十分」を数学的・客観的な保証とせず、次の意味として扱う。

> 現時点の目的・前提・リスク受容に照らし、責任者が受入れ可能と定めた品質条件。

検証時には、少なくとも次を観測する。

- Key-Quality通過率
- 人間の修正時間
- 判断回数
- 手戻り回数
- 中断・疲労の自己申告
- 総コスト

#### Terra対応

- 案件別の受入条件テンプレート: 対応可能
- 観測指標の集計: 対応可能
- 許容量の決定: 人間判断が必要

---

### P1-2. Handoff後の前提変更検知が自己参照している

「前提が変われば再評価する」という原則は妥当である。

しかし、人間が実行の手を離した後に、誰がどのように変化を検知するかが確定していない。

監視を増やしすぎると、Handoffによって減らすはずの人間負荷が再び増える。

#### 検証対象

- 検知率
- 見逃し率
- 誤検知率
- 検知遅延
- 再評価に要した時間
- 監視自体のコスト

#### Terra対応

- 前提変更テストケース作成: 対応可能
- 文書、モデル版、入力形式などの変更検知ルール案: 対応可能
- 許容する見逃しリスクと監視予算の決定: 人間判断が必要

---

### P1-3. 5軸と操作プリミティブは独立していない

自然言語上、次の分類は重なりやすい。

- IntentとPosition
- BoundaryとConstraint
- DirectionとMagnitude
- BoundaryとFilter
- PriorityとSequence

5軸を独立した直交次元のように扱うと、単独効果を測れない。

現状の「操作上の分類」という位置づけを維持し、相互作用することを明記する必要がある。

#### Terra対応

- 重複表現・役割の比較表作成: 対応可能
- 各分類を独立軸とするか、非直交な操作分類とするか: 人間判断が必要

---

### P1-4. Approach Failureが広すぎる

現行のApproach Failureには、次が混在している。

- ハルシネーション
- 不可侵領域違反
- 目的・形式からの逸脱
- 前提変更
- モデル能力不足
- 安全装置による拒否

原因と症状が同じラベルへ入っているため、Retry時にどこを直すべきか判別しにくい。

#### 分類案

| 分類 | 内容 |
|---|---|
| Axis操作失敗 | Axisを変えても意図方向へ動かない |
| 内容品質失敗 | 事実誤認、計算ミス、根拠不足 |
| 境界失敗 | 不可侵領域・禁止事項への違反 |
| 運用失敗 | 前提変更や版変更を検知できない |
| 能力不足 | 対象モデル・ツールでは実行できない |
| 正常な拒否 | 安全境界が正しく機能した |

Approach Failureを上位停止ラベルとして残す場合でも、原因分類を別に記録する。

#### Terra対応

- 現行の失敗例を分類案へマッピング: 対応可能
- 正式なFailure分類の採否: 人間判断が必要

### P1-5. Advisorが軸を保持せず、誤解を増幅する可能性

AdvisorとExecutorが同じ誤解、文体、評価バイアスを共有する場合、人間の確認負荷を下げながら誤りを安定化する可能性がある。

Advisor構造は、次を分けて比較する必要がある。

- Advisorなし
- Executor自身による自己評価
- 同一系列の別モデルによる評価
- 別系列モデルによる評価
- 人間によるブラインド評価

#### 検証指標

- 軸保持率
- 誤りの見逃し率
- 人間レビュー時間
- 再作業率
- 前提変更検出率
- モデル費用
- 総コスト

#### Terra対応

- 比較プロトコル作成: 対応可能
- 結果集計: 対応可能
- 自身をAdvisorとして用いた検証の客観性確保: 別評価者が必要

## 3. 今後の検証工程

### Phase 0. Claim Ledgerを作る

本文中の主張を、次へ分類する。

| 種別 | 意味 |
|---|---|
| 定義 | Cokes / Axis内でそう定める |
| 規範 | そのように運用すべきとする |
| 比喩 | 人間が操作を理解するための説明 |
| 作業仮説 | 観測・反証の対象 |
| 外部研究による参考 | 類似した問題設定や説明の足場 |
| 実証済み観測 | 条件、モデル、版、実験記録を伴う観測 |

この分類がないまま実験すると、比喩を証明しようとしたり、定義を実証結果のように扱ったりする。

### Phase 1. Axisの最小反証実験

文章タスクから開始する。

#### 比較条件

1. 通常プロンプト
2. Axisで構造化したプロンプト
3. 情報量・長さだけ増やした疑似Axis
4. 意図的に強すぎるAxis
5. 意味を保った複数の言い換え

#### 観測指標

- 事前定義した方向へ寄ったか
- Key-Quality通過率
- 指示していない部分まで変わった割合
- 人間の修正時間・修正回数
- 評価者間一致
- モデル費用
- 総コスト

内部の操作量を数値化しない方針と、出力効果を数値で観測することは両立する。

### Phase 2. Cokes工程比較

同一のKey-Outputについて、次を比較する。

- 人間のみ
- 通常のAI利用
- Axisを使ったAI利用
- AI以外のSaaS、コード、既存手段
- Advisor + Executor

Cokesの勝利条件は、AI出力が良くなることだけではない。

> Key-Qualityを維持しながら、人間負荷と総コストが下がること。

### Phase 3. Handoff耐久試験

意図的に前提変更を注入する。

- 正本用語の変更
- 入力形式の変更
- モデル版の変更
- 責任者の変更
- 後工程の受入条件変更
- 検知しにくい小さなドリフト

変更を検知できたか、何件見逃したか、再評価までどの程度かかったかを測る。

### Phase 4. Advisor構造の比較

Advisorなし、同一モデル自己評価、別モデルAdvisor、人間評価を比較する。

人間レビューが減っても、見逃し率、再評価遅延、総コストが増える場合は不採用とする。

### Phase 5. 実務案件での外的妥当性確認

小規模実験で有効でも、実務で同じ効果が出るとは限らない。

異なる種類の案件へ展開し、モデル差、担当者差、期間差、前提ドリフトを観測する。

## 4. 画像生成へのAxis検証

現行の第10章 §10.5は、Axisの検証結果ではなく、ControlNet、LoRA、Prompt Schedulingを用いた制作順序の作業仮説である。

次を区別する必要がある。

1. 自然言語プロンプトだけによるText-to-Image
2. 参照画像を用いたImage-to-Image / Editing
3. ControlNet等による構造制御
4. LoRA等による追加学習・スタイル制御
5. Prompt Scheduling等のツール固有制御

これらを一つの「画像生成Axis効果」としてまとめない。

### 画像上で観測するAxis

| Axis | 画像での観測例 |
|---|---|
| Intent | 広告、記録写真、教材、物語挿絵などの用途 |
| Abstraction | 写真、イラスト、図解、アイコンなどの表現粒度 |
| Boundary | 個数、色、位置、禁止要素、文字の有無 |
| Priority | 人物、商品、背景、構図、文字のどれを優先したか |
| Direction | 温かい、緊張感、静かな、批判的などの方向 |
| Sequence | 語順・提示順を変えた場合の感度 |

Sequenceは画像意味の独立軸ではなく、入力順序効果として別扱いにする可能性がある。

### 評価項目

- プロンプト忠実度
- 構図、個数、位置、色
- 意図したAxis方向
- 美的品質
- 指示していない要素の変化
- 生成失敗率
- 人間の修正時間
- 総コスト

「いい画像になった」だけでは、Axis効果とモデルの美的バイアスを区別できない。

### 最初の小規模試験案

```text
2モデル
× 10課題
× 4プロンプト条件
× 4生成
= 320画像
```

プロンプト条件の例:

1. 通常プロンプト
2. Axis構造化プロンプト
3. 意味を保った言い換え
4. 過剰に強めたAxis

評価時にはプロンプト種別・モデル名を隠し、A/Bの順序をランダム化する。

seedを固定できる場合は、固定seedによる差分比較と、ランダムseedによる安定性比較を分ける。

### 参考となる評価研究

- TIFA: 質問応答を用いた細粒度なText-to-Image忠実度評価  
  <https://arxiv.org/abs/2303.11897>
- GenEval: 物体、位置、個数、色などの評価  
  <https://arxiv.org/abs/2310.11513>
- Gecko: プロンプトの曖昧さ、人間評価テンプレート、評価指標の信頼性  
  <https://arxiv.org/abs/2404.16820>
- HEIM: 忠実度、美観、頑健性、公平性、効率などの多面的評価  
  <https://arxiv.org/abs/2311.04287>

画像生成における主要な検証命題は次である。

> Axis操作によって狙った視覚特性が動き、Key-Qualityと他の重要特性が維持されるか。

## おまけ：この「ポエム」の現代での役割

### 人間のつぶやき

> このポエムについて、現代での役割はどこにあると思う。

### Solの応答

この「ポエム」の現代での役割は、**AIをどう動かすかではなく、人間が何を望み、どこまで任せ、何を背負うかを言語化すること**にある。

現在のAIをめぐる議論には、それぞれ個別の領域がある。

- プロンプト技法は「どう指示するか」
- エージェント／ハーネスは「どう実行させるか」
- 評価基盤は「どう採点するか」
- AIガバナンスは「何を禁止・管理するか」
- 組織論は「誰が責任を持つか」

しかし、これらを人間の実務感覚で一続きにする言葉は、まだ弱い。

Cokes + Approach Axisが埋めようとしているのは、その間である。

> 何をいいモノとするのか。  
> どの方向へ寄せたいのか。  
> どこから先は任せられるのか。  
> 任せた後も、誰が責任を持つのか。

これはモデル理論というより、**人間が確率的な道具と共同作業するための運用思想**である。

特に現代的なのはHandoffである。AIの性能競争が進むほど、問題は「できるか」から「人間が確認し続けなくても任せられるか」へ移る。

そこでCokesは、次の線引きを置こうとしている。

- Key-Qualityを満たしている
- 人間負荷が破裂しない
- 前提変更を検知できる
- 最終責任は人間が保持する

ここは単なるポエムではなく、実務上かなり鋭い。

Approach Axisの地形・砂・磁石も、AI内部機構の説明として扱えばポエムである。しかし、複雑なモデル挙動を人間同士で話すための**共有比喩**として価値がある。

人間は高次元の内部表現を直感的には扱えない。そのため、「少し寄せる」「境界を置く」「傾斜が強すぎて崩れた」という操作言語が必要になる。

このポエムの役割は、三つに整理できる。

1. **価値観を圧縮する憲法**  
   「楽して、いいモノを作る。ただし責任は捨てない」
2. **人間とAIの間を話す共通言語**  
   意図、境界、強さ、失敗、Handoffを会話可能にする
3. **検証仮説を生む母体**  
   「この軸は本当に効くか」「人間負荷は減るか」を実験へ落とす

逆に、これを学術理論やAI内部の真相として扱い始めると壊れる。

最も収まりのよい立ち位置は、次である。

> **Cokesは実務の憲法、Approach Axisは操作の仮説言語。検証結果が、そのポエムを少しずつ工学へ変えていく。**

ポエムは、工学になる前に捨てるものではない。人間が「何を測るべきか」を決めるために必要な、工学より一段上の起点である。

数値だけでは、「楽して、いいモノ」の「いい」を決められない。
