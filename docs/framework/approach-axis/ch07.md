# 第7章: 実行フローと状態モデル（実務パターン）

> 本文 v1 / 2026-07-23（レビュー済み・昇格済み）
> 用語の正本: `glossary.md`

第5章では軸語彙を、第6章では操作プリミティブを整理した。本章では、それらを人間とAIの作業手順へ接続する。ここで扱うのはAI内部の処理手順ではなく、設計者が「何を決め、何を試し、どの結果を次の操作へ渡すか」を管理するための実務フローである。

本章のフローは固定手順ではない。目的・リスク・可逆性によって操作順序を選び、出力の観測結果に応じて途中で切り替える。公開プロンプト資料は、Axisの正しさを証明するものではなく、実務上の操作パターンを観測・比較するための参考資料として扱う。

## 7.1 共通の実行フロー

どの作業でも、次の5段階を基本単位とする。

```text
マッピング → プランニング → 実行 → Axis評価 → Cokes判定 → 記録とHandoff
                 ↑                    │             │
                 └── Retry / 再設計 ───┘             └─ Lock
```

### 1. マッピング

人間の軸レンジとAI Profileを、作業に必要な範囲だけ確認する。

- **人間の軸レンジ:** 目的、方向性、Boundary、優先順位、許容範囲
- **AI Profile:** そのモデルで観測された反応傾向、得意な粒度、逸脱しやすい条件
- **入力の地形:** 既存文書、用語集、前提、候補、未確定事項

AI Profileはモデル内部を測定したプロフィールではない。過去の出力や比較試行から、人間が再利用するために記録した観測仮説である。未知の項目は空欄のままにし、推測で埋めない。

### 2. プランニング

軸語彙、操作プリミティブ、受入条件を自然言語で決める。ここでは内部の座標値や重みを指定しない。

```text
Position: 「初学者にも追える概念説明へ寄せる」
Magnitude: 「少し広げる。主題と構成は維持する」
Constraint: 「用語集の定義を変更しない」
Filter: 「確認できない外部情報は事実として扱わない」
受入条件: 「主題を保ち、用語の定義が正本と一致している」
```

受入条件は、後でSuccessを判定できる程度に具体化する。ただし、全タスクに共通する数値閾値や万能な品質スコアは置かない。

### 3. 実行

計画したプリミティブを自然言語の指示として適用し、出力を取得する。複数案が必要ならBlendを使い、前提や境界を変更した場合はその変更を記録する。

### 4. Axis評価

出力が指定方向へ寄ったか、Constraint・Filterに違反していないか、Axis上の受入条件を満たすかを人間が確認する。内部でどのような表現変化が起きたかは判定材料にしない。ここで判定できるのは、あくまで「Axisの操作として成功したか」である。

### 5. Cokes判定と記録・Handoff

Axis評価がSuccessでも、直ちにHandoffできるとは限らない。Cokesの上位視点から、品質維持が確認され、人間が担っていた工程を、明示した前提の範囲でAIまたは次工程へ任せられるかを判定する。Cokes判定を通過したらLockして委任し、未通過ならAxisへ差し戻すか、研究層・人間レビューなど別の確認へ回す。

Handoffは、次の2段階を通過した成果物に対して発行する。

1. **Axisゲート:** 指定方向、Constraint・Filter、Axis上の受入条件を満たす。
2. **Cokesゲート:** 品質維持が確認され、人間が担っていた工程、任せられる範囲、上位体系との整合、責任の所在、前提変更時の再評価条件が確認できる。

Axisの内部操作とCokesの上位判定を混同しない。Cokes判定はAIの内部状態を判定するものではなく、品質が維持された工程を、明示した前提の範囲で人間からAIまたは次工程へ委任してよいかを人間が確認するゲートである。

「任せられる」と「任せる」は別の判断である。Cokesは委任可能性を評価できるが、実際に委任する決定と責任の引き受けは人間が行う。委任しない、または人間の監視を残すという選択も、Handoff設計に含まれる。

## 7.2 操作順序の2パターン

5段階の枠組みは共通だが、Constraintと探索の順番は作業によって異なる。ここでは代表的な2パターンを置く。

### 制約先行型：絞ってから作業する

```text
Position → Constraint → Magnitude → Filter → Retry / Lock
```

安全境界、対象範囲、禁止事項を先に明示し、その内側で出力を作る。次の作業に向く。

- 不可逆な処理や外部への公開を伴う
- 対象と目的が明確で、逸脱コストが高い
- 確認可能な事実だけを扱う必要がある
- 変更範囲を狭く保つこと自体が価値になる

**例:** 用語集を正本として章ドラフトを整える場合、「glossary.mdの定義を変更しない」「外部参照をAxisの証明と書かない」を先にConstraintへ置く。

### 探索先行型：可能性を残してから絞る

```text
Position → Magnitude → Blend → Filter → Constraint → Retry / Lock
```

安全上の不可侵領域は確保したうえで、候補・観点・表現を広げてから、不要なものをFilterとConstraintで絞る。次の作業に向く。

- 創作や仮説検討で、初期に候補を狭めると発見を失う
- 複数案の比較に価値がある
- 失敗しても前の状態へ戻せる
- 最終的な表現や粒度を後から決められる

**例:** 章の構成案を複数出す場合、最初から一案に固定せず、異なるPositionやMagnitudeの案をBlendして比較し、採用後に用語・境界・形式をConstraintで確定する。

探索先行型でも、不可侵領域や安全境界を後回しにしてよいわけではない。後段へ回せるのは、候補の範囲や表現上の制約である。2つのパターンは固定手順ではなく、途中で切り替え可能な運用パターンとする。

## 7.3 パターンを選ぶ基準

開始時に、次の観点を短く記録してパターンを選ぶ。

| 観点 | 制約先行型へ寄せる条件 | 探索先行型へ寄せる条件 |
|---|---|---|
| 目的 | 到達点が明確 | 到達点を比較しながら決める |
| リスク | 逸脱の影響が大きい | 逸脱を人間が確認できる |
| 可逆性 | 戻しにくい | 以前の状態へ戻せる |
| 候補の価値 | 候補を増やす必要がない | 多様な候補が成果に寄与する |
| 確認コスト | 早期に範囲を絞る方が安い | 候補生成後にまとめて確認できる |

選択理由そのものも、後で比較できる運用記録である。実行中に目的やリスクが変わったら、同じパターンを惰性で続けず、別のパターンへHandoffする。

## 7.4 公開プロンプト集を解析する

OpenAI Academyの「ChatGPT for any role」のような公開プロンプト資料は、Axisの検証用サンプルとして読める。ページは、用途、プロンプト例、ChatGPTで試す導線を組み合わせたPrompt Packとして構成されている。[OpenAI Academy「ChatGPT for any role」](https://academy.openai.com/public/clubs/work-users-ynjqu/resources/chatgpt-for-any-role)

### 解析の手順

1. **観測事実を写す**
   用途名、プロンプトの文、差し込み変数、出力形式、実行導線を記録する。公開資料にない内部事情は補わない。
2. **構造を分解する**
   目的・対象・文脈・制約・出力形式・評価観点に分ける。
3. **Axisへ仮分類する**
   目的や対象をIntent / Position、強さや粒度をMagnitude、禁止事項や形式をConstraint、選別条件をFilter、複数候補の扱いをBlendとして読む。分類は仮説であり、用語集の定義を変更しない。
4. **一致しない部分を残す**
   Axisで説明しにくい構造は、用語のズレ、Axis側の不足、用途固有の実務知として記録する。
5. **同一課題で比較する**
   元のプロンプトと、Axisで再構成したプロンプトを同じ入力で試す。受入条件、出力の安定性、修正回数、人間の扱いやすさを比較する。
6. **結果を反映する**
   Axis版が有用なら操作パターンへ取り込む。元の構造が有用なら、Axisの説明を修正する。どちらも成立しない場合はタスク固有の例として保留する。

### 解析例：対象読者に合わせた書き換え

「対象読者に合わせてメッセージを再構成する」タイプのプロンプトは、次のように読める。

| 公開プロンプトの要素 | Axisでの仮分類 | 確認する出力 |
|---|---|---|
| 読者を指定する | Position / Intent | 読者に必要な観点へ寄っているか |
| 元の文脈を渡す | 入力の地形 | 元の目的・前提を保っているか |
| 明確・丁寧・簡潔にする | Magnitude / Constraint | 目標の文体と粒度になっているか |
| 本文を差し込む | 対象範囲 | 入力外の内容を勝手に補っていないか |

この分類は、OpenAI Academyの内部設計を説明するものではない。公開されたプロンプトの働きを、Axisの語彙で操作可能な単位へ読み替えた作業仮説である。

### 一致しないときの再整理

Axisと公開資料の構造が一致しないとき、どちらかを先に正しいと決めない。次の順番で再整理する。

- **実務上の有用性:** その構造を使うと、受入条件を満たす出力が安定して得られるか
- **記録のしやすさ:** 人間が変更点と結果を比較できるか
- **再利用性:** 別の課題やモデルへ移しても操作として理解できるか
- **説明の妥当性:** 観測事実と解釈モデルが混ざっていないか

公開資料は参考となる公開資料であり、Axisの証明ではない。同時に、Axisの理屈と一致しないからといって、実務上の有効性を捨てる理由にもならない。

## 7.5 状態モデル

実行フローが「人間の作業順序」を示すのに対し、状態モデルは「評価結果を受けて次に何をするか」を示す。

```text
Idle
  ↓
Plan
  ↓
Execute
  ↓
Evaluate
  ├─ Axis Success ─────→ Cokes判定
  │                         ├─ 通過 → Lock → Handoff
  │                         └─ 未通過 → Axisへ差し戻し / 別層確認
  ├─ Partial Success ──→ Retry / 再アプローチ計画
  └─ Approach Failure ─→ 停止・原因分析・再設計
```

### Idle

作業対象、責任者、目的がまだ確定していない状態。入力や受入条件が曖昧なままExecuteへ進めない。

### Plan

軸語彙、操作プリミティブ、操作順序、受入条件、必要な参照資料を決める。探索先行型を選んだ場合でも、安全境界はこの状態で確認する。

### Execute

計画した指示を適用し、出力と変更点を記録する。1回の実行で結論を固定せず、必要なら候補を残す。

### Evaluate

人間が受入条件、Constraint、Filter、意図方向への接近を確認し、Axisの判定ラベルを付ける。Axis Success後に、Cokesの上位判定を行う。

## 7.6 判定ラベルと次アクション

| 判定 | 条件 | 次アクション |
|---|---|---|
| **Axis Success** | 受入条件を満たし、意図方向への重大な逸脱がなく、Constraint・Filterを満たす | Cokes判定へ進む |
| **Handoff可** | Axis Successに加え、人間が定義した評価軸に沿う品質が維持され、明示した前提が変わるまでは対象工程を任せられる。委任範囲と再評価条件が確認できる | LockしてHandoff |
| **Partial Success** | 一部は有用だが、変更箇所を特定して修正できる | 変更する軸またはプリミティブを1つ以上選びRetry |
| **Approach Failure** | 重大な逸脱、検証不能な主張、不可侵領域違反、または方向の完全な不一致 | 停止し、原因仮説・Boundary・入力地形を再確認 |

Axis Successは、モデル内部が望ましい状態になったという意味ではない。指定したAxis上の受入条件を出力が満たしたという意味である。Handoff可は、さらにCokesの上位判定を通過し、人間が定義した評価軸に沿う品質が維持され、人間が担っていた工程を前提が変わるまでは任せられると判断した状態である。

Partial Successでは、毎回すべてを変えない。Position、Magnitude、Constraint、Filter、Blend、順序のうち、どれを変更したかを記録する。Approach Failureでは、Retryを無制限に続けず、入力不足・Boundaryの誤設定・Positionの誤認・AI Profileの観測不足など、原因仮説を更新する。

## 7.7 記録とHandoff

Handoffで渡すのは成果物だけではない。その成果物が成立した前提条件、変更履歴、未確定事項、次工程での再評価条件も一緒に渡す。前提が確定していない状態でのHandoffは、次工程へ不明なリスクを移すだけである。

人間による検証も、定義済みの受入条件・監視・再評価条件もないままAI出力を次工程へ通過させることは、Cokes側の責務放棄とする。一方、個別目視は、これらの条件と委任範囲が記録され、「任せられる状態」が確認できる場合に限って委任できる。

### 前提条件の確定

最低限、次の前提を確認してからCokes判定へ進む。

- 目的・Intentと対象範囲が確定している
- 参照した資料、用語集の版、入力地形が記録されている
- AI Profileの観測範囲と未確認事項が区別されている
- Constraint・Filter・受入条件が明示されている
- 人間の責任者と次工程が決まっている

前提が「仮置き」のままでも作業は開始できるが、その場合はHandoff条件に仮置きであることと確認期限を含める。確定前の成果物を確定済みとしてLockしない。

### 前提変更時の再評価

前提に変更があった場合、変更の大きさによって扱いを分ける。

| 変更の種類 | 扱い |
|---|---|
| 表現、粒度、局所的な例の変更 | 変更箇所を記録し、必要な範囲でAxis評価をRetryする |
| 目的、対象範囲、Boundary、正本、根拠の変更 | 既存LockとHandoffを一度無効化し、Planから再評価する |
| 責任者、次工程、Cokes上位方針の変更 | Cokes判定をやり直し、新しいHandoffを発行する |

前提変更後に旧判定をそのまま引き継がない。新しい前提は新しいチェックポイントとして扱い、Axis評価とCokes判定の両方を必要な範囲で再実施する。

### Handoff記録

最低限、次の項目を運用カードや作業ログに残す。

```text
対象:
目的 / Intent:
委任する工程:
任せられる範囲:
前提条件:
前提の確認状態: 確定 / 仮置き / 要再確認
前提の変更履歴:
Position:
操作順序: 制約先行型 / 探索先行型
Magnitude:
Constraint:
Filter:
入力資料・前提:
受入条件:
出力の観測:
判定: Success / Partial Success / Approach Failure
Cokes判定: 通過 / 差し戻し / 別層確認
Handoff条件: 次工程で何をする / 何をしない
再評価条件:
品質維持の観測: 軸保持率 / 修正率 / レビュー圧縮 / 認知負荷など
変更した軸・プリミティブ:
未解決事項:
次の責任者 / Handoff先:
```

Success後のLockは、人間が合意した範囲・用語・出力を次工程で不用意に変更しないための操作である。前提が変わった場合は、既存のLockを黙って破らず、Lockを無効化して再評価し、新しいHandoffとして前提変更を記録する。

## 7.8 機構非依存のガードレール

- 実行フローはAI内部の処理手順ではなく、人間の設計・評価・記録の手順である。
- プロンプト操作と内部アクティベーションへの直接介入が、同じ内部表現変化を生むとは仮定しない。
- 公開プロンプト資料は、参考となる公開資料・比較対象であり、Axisの直接的な証明ではない。
- 軸の効力順位、モデル別傾向、公開プロンプトの有効性は、タスク依存の作業仮説として検証する。
- 成否は隠れた内部状態ではなく、出力・受入条件・人的レビューで判定する。

## 7.9 まとめ

- 5段階フローは、マッピング、プランニング、実行、Axis評価、Cokes判定と記録・Handoffで構成する。
- 操作順序には、制約先行型と探索先行型があり、目的・リスク・可逆性に応じて使い分ける。
- 公開プロンプト集は、Axisの語彙で分解し、元の構造との比較から実務上の有用性を検証できる。
- Axisと一致しない有効な構造を見つけたら、Axisへ無理に合わせず、用語・構造・有効性を再整理する。
- Axis評価は操作の成否を判定し、Cokes判定は品質維持を確認したうえで人間の工程を前提の範囲で任せられるかを判定する。両方を通過して初めてLockとHandoffへ進む。

## 用語集との対応

本章で新しい正本用語は追加しない。`Position`、`Magnitude`、`Constraint`、`Filter`、`Blend`、`Retry`、`Lock`、`Handoff`、`Success / Partial Success / Approach Failure`は既存の定義を参照する。OpenAI Academyは「外部参照・関連概念」にある「参考となる公開資料」として扱う。
