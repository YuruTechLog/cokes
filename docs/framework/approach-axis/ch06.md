# 第6章: 操作プリミティブ7種

> 本文 v1 / 2026-07-23（レビュー済み・昇格済み）  
> 用語の正本: `glossary.md`

第5章では、何をどの方向へ寄せるかを軸語彙で設計した。本章では、その設計を実際の運用へ渡すための基本動詞を整理する。

ここでいう操作プリミティブは、AI内部の関数やAPIではない。人間が「何を変えたか」を記録し、次の試行で再利用するための操作単位である。コマンド例はすべて擬似コードであり、特定のLLMや実装がこの命令を直接受け付けることを意味しない。

## 6.1 7種の全体像

| プリミティブ | 日本語の役割 | 何を操作するか | 擬似コード例 |
|---|---|---|---|
| Position | 位置づけ | 人間の軸レンジに対して、出力をどこへ寄せるか | `setPosition(target: conceptual_explanation)` |
| Magnitude | 強度調整 | 寄せ方の強さ・広がりをどのようなニュアンスで指定するか | `setMagnitude(tone: slight)` |
| Constraint | 制約 | 不可侵領域・禁止ルール・対象外を設定する | `applyConstraint(ConstraintZone, enforce)` |
| Filter | 濾過 | 生成された候補から、条件に合わないものを除く | `applyFilter(factualOnly)` |
| Blend | 混合 | 複数案の要素を組み合わせる | `blendOptions(primary: A, secondary: B, balance: dominant)` |
| Retry | 再接近 | 失敗や部分成功を受け、条件を変えて再試行する | `retryApproach(strategy: reviseBoundary)` |
| Lock | 固定 | 合意した定義・範囲・出力を次の工程で固定する | `lockAxis(topic: summary)` |

7種は同じ種類の操作ではない。PositionとMagnitudeは寄せ方を設計し、ConstraintとFilterは逸脱を抑え、BlendとRetryは複数の候補を扱い、Lockは合意後の変更を止める。それぞれの責務を分けることで、失敗したときに「何を変えるべきか」を追跡できる。

## 6.2 Position：位置づけ

Positionは、出力を人間の軸レンジのどの位置へ寄せるかを決める。ここでいう位置はAI内部の座標を意味しない。読者、目的、観点、表現の寄せ先を、人間が操作上の位置として記録する。

```text
Positionを決める
  → 「概念説明」「実務手順」「批判的検討」などの寄せ先を明示する
  → 出力を確認する
```

**例:** 「概念説明の位置に寄せ、実装コードの提案までは進めない」と指定する。

`setPosition(target: conceptual_explanation)` は、現在の出力からどの意味のランドマークへ寄せるかを指定する表記である。内部座標やベクトル値を指定するものではない。

## 6.3 Magnitude：強度調整

Magnitudeは、Positionや軸語彙による寄せ方をどの程度強くするかを調整する。強くするほど目的方向への集中が期待できる一方、境界を越えたり、目的外の意味まで巻き込んだりする危険が増える。

```text
強度を上げる → 目的方向へ集中しやすい／巻き込みが増える
強度を下げる → 余白が残る／目的がぼやける
```

**例:** 「方向性を一度に強めず、まず `slight` で出力を確認し、必要な場合だけ `moderate` へ寄せる」と段階を決める。

操作量は、万人に共通する物理量ではなく、意味の変化を人間同士で共有するためのニュアンス語彙で表す。第6章では次の語を初期の管理語彙として使う。

| ニュアンス語 | 日本語の目安 | 代表的な指示 | 操作上の意味 | 受入条件の目安 |
|---|---|---|---|---|
| `slight` | 少し／軽く | 「少し広げて。元の主題と構成は維持して」 | 隣接する観点を加える、または粒度を少し動かす | 主題・構成を保ったまま、隣接する観点や小さな粒度変更が確認できる |
| `moderate` | 中程度 | 「関連する複数の観点を加えて、説明の組み立ても調整して」 | 複数の観点を加える、または説明の高さを明確に切り替える | 複数の観点または粒度の切替があり、指定した境界内に収まっている |
| `strong` | 強く／大きく | 「前提を疑い、反対側の視点から全体を組み替えて」 | 中心的な見方や前提まで組み替える。Filter・Constraintを併用する | 見方の転換が出力に現れ、明示したConstraintを破らない。人間が最終確認する |

ここでの受入条件は、モデル内部の変化を測る数値ではなく、出力を人間が確認するための観測基準である。ニュアンス語は目安であり、タスク固有の受入条件を優先する。

会話上の `a little` は `slight` の言い換えとして使えるが、`little` は「ほとんどない」という意味にもなりうるため、管理ラベルには採用しない。`big` もサイズの意味に寄りやすいため、操作量には `strong` を使う。

Magnitudeはモデル内部の重みを直接変更する操作ではない。プロンプトの表現、反復、境界の明示、提示順など、人間が調整できる条件の強さを、`slight / moderate / strong` のようなニュアンス語で記録する。

## 6.4 Constraint：制約

Constraintは、AIに踏み込ませない不可侵領域、禁止ルール、対象外の範囲を明示する。第5章のBoundaryと関係するが、Boundaryが「範囲をどう定めるか」という軸語彙であるのに対し、Constraintはその範囲を運用上の制約として適用するプリミティブである。

**例:** 「確認できない外部情報を事実として補わない」「用語集の定義を変更しない」を不可侵領域として設定する。

Constraintを置いても、AI内部に完全な壁が生成されるわけではない。出力に制約違反がないかは、Filterと人間の評価で確認する。

## 6.5 Filter：濾過

Filterは、生成された候補を条件に照らして選別する。Constraintが事前に範囲を定めるのに対し、Filterは出力後に候補を確認し、残すものと戻すものを分ける。

| Filterの観点 | 確認例 |
|---|---|
| 根拠 | 参考資料にない断定を含んでいないか |
| 範囲 | 指定した対象外へ広がっていないか |
| 粒度 | 概念説明なのに実装詳細へ落ちていないか |
| 用語 | glossary.mdの正本と整合しているか |

**例:** `applyFilter(factualOnly)` は、確認できた事実と推測・比喩を分け、事実として残す内容を選別する操作を表す。創作や仮説検討では、「確認可能な範囲に限定する」「推測であることを表示する」など、目的に合ったFilterを選ぶ。

## 6.6 Blend：混合

Blendは、複数の候補や観点から、採用する構成を組み合わせる。単一案を無理に強めるのではなく、案Aの構造と案Bの具体例を混ぜるなど、複数の部分成功を人間が統合するために使う。

**例:** 「案Aの用語の正確さを残し、案Bの初心者向け例を組み合わせる」と決める。

`blendOptions(primary: A, secondary: B, balance: dominant)` は、採用した要素の主従を自然言語で記録するための擬似的な表記である。出力の品質を自動的に数値合成する仕組みではない。混合後の案は、人間が再度レビューする。

## 6.7 Retry：再接近

Retryは、同じ指示を無制限に投げ直すことではない。Partial SuccessやApproach Failureの原因仮説を立て、変更する条件を記録したうえで再接近する。

```text
Evaluate
  ├─ Success → Lock
  ├─ Partial Success → 一つ以上の条件を変更してRetry
  └─ Approach Failure → 停止・根拠トレース・制約再確認 → 再設計
```

**例:** 「抽象度は維持し、境界だけを狭めて再接近する」と変更点を限定する。

Retryでは、毎回すべてを変えない。起点・方向性・境界・抽象度・優先順位・順序のうち何を変更したかを自然言語で残すことで、結果の差を比較できる。

再接近には終了条件を置く。たとえば「指定方向に寄り、Constraint違反がない状態になったら止める」「変化が見られなければ別の軸を見直す」「前提が変わったらRetryではなく新しいHandoffに戻す」といった条件である。終了条件のない再接近は、リトライ沼になる。

## 6.8 Lock：固定

Lockは、合意した範囲、用語、出力、または判断を次の工程で変更しない状態にする。LockはAIを固定する命令ではなく、人間側の合意を固定する運用である。

**例:** 「第4章の4層名と役割を確定し、第5章以降では変更しない」と記録する。

Lock後に前提を変える必要が生じた場合は、既存のLockを黙って破らず、新しいHandoffとして前提更新を記録する。

## 6.9 7種を一つの流れで使う

```text
Position   ── 寄せ先を決める
Constraint ── 不可侵領域を置く／確認する
Magnitude ── 寄せ方のニュアンスを調整する
Filter     ── 出力を確認・選別する
Blend      ── 複数案を組み合わせる
Retry      ── 必要なら条件を変えて再接近する
Lock       ── 合意を固定してHandoffする
```

すべてのプリミティブを毎回使う必要はない。目的と状態に応じて必要なものだけを選ぶ。操作順には少なくとも二つの型がある。

### 制約先行型：絞ってから作業する

安全性、不可逆な処理、対象外を明確にすべき作業では、先にConstraintを置く。

```text
Position → Constraint → Magnitude → Filter → Retry / Lock
```

### 探索先行型：可能性を残してから絞る

創作、仮説生成、複数案の比較など、最初から候補を狭めたくない作業では、Constraintを最小限にして候補を広げ、その後にFilterやConstraintで絞る。

```text
Position → Magnitude → Blend → Filter → Constraint → Retry / Lock
```

探索先行型でも、不可侵領域を後回しにしてよいわけではない。安全上の境界は先に置き、候補の範囲や表現上の制約だけを後段で絞る。上の二つは固定手順ではなく、目的・リスク・可逆性に応じた運用パターンである。

## 6.10 機構非依存のガードレール

- プリミティブはAI内部の関数やAPIではなく、人間の運用上の操作単位である
- `Position` や `Magnitude` は、内部座標や重みを直接変更する命令ではない
- Activation Steering等の研究は、出力制御を考えるための参考となる学術理論・研究上の視点であり、同一実装の証明ではない
- 擬似コードの引数は、内部状態の測定値や品質スコアではない。操作のニュアンスと変更内容を記録するための自然言語ラベルである
- 成否は隠れた内部状態ではなく、出力と人間のレビューで判定する

この章でのPosition単体の勝敗は、内部でどのような表現変化が起きたかでは決まらない。**指定した方向に出力が寄っていれば、Position操作は成功である。** ただし、全体のSuccess判定には、明示したConstraint・Filter・人間の受入条件も含める。中身が見えなくても、結果を観測し、次の操作へ戻せればよい。

## 6.11 まとめ

- 操作プリミティブは、軸語彙で設計した内容を運用上の動詞へ変換する
- Position・Magnitudeは寄せ方、Constraint・Filterは逸脱抑制、Blend・Retryは再設計、Lockは合意固定を担う
- 擬似コードは実装仕様ではなく、自然言語の操作ニュアンスと変更内容を記録するための表記である
- 7種は一方向の固定手順ではなく、評価結果に応じて循環的に使う
